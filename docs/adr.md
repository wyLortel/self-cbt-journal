## ADR-001: 프론트엔드 프레임워크 선택 (React + TypeScript + Vite)

**Status**: Accepted  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

프론트엔드 개발 프레임워크를 선택해야 한다. 대학생을 타겟으로 한 감정 기록 앱이므로:

- **개발 속도**: 4주 MVP 출시가 목표
- **커뮤니티**: 초기 버그 해결과 라이브러리 선택지 중요
- **타입 안전성**: 감정 데이터 스키마의 정확성 필수
- **개발 경험**: 빠른 피드백 루프 (hot reload)

### Decision

**React 18+ + TypeScript + Vite를 선택한다.**

프론트엔드 개발 속도, 커뮤니티 크기, 타입 안전성의 조합을 고려했을 때 React가 최적이며, 빌더로는 Vite를 사용해 개발 속도를 극대화한다.

### Alternatives

#### 1. Vue.js + TypeScript

**장점:**
- 러닝커브가 낮음 (템플릿 문법이 HTML에 가까움)
- 공식 라우터, 상태관리(Pinia) 통합
- 문서가 명확함

**단점:**
- 커뮤니티 규모 작음 (라이브러리 선택지 제한)
- 회사 채용 시 Vue 경험자 찾기 어려움
- UI 컴포넌트 라이브러리 생태계 약함

**결정 이유:** 초기 개발 속도와 추후 팀 확장을 고려하면 React의 생태계가 월등함

#### 2. Angular

**장점:**
- 완전한 풀스택 프레임워크 (라우팅, HTTP, 폼 내장)
- 강력한 타입 시스템
- 대규모 엔터프라이즈 프로젝트에 최적

**단점:**
- 학습곡선이 매우 높음 (초기 생산성 저하)
- 번들 사이즈 크고 성능 오버헤드
- MVP 단계에서 과도한 기능

**결정 이유:** 초기 시작 단계이므로 오버엔지니어링

#### 3. Svelte

**장점:**
- 컴파일 타임 최적화로 번들 사이즈 매우 작음
- 반응성(reactivity)이 직관적
- 높은 성능

**단점:**
- 커뮤니티 규모 매우 작음
- 라이브러리 생태계 부족
- 회사 채용 어려움

**결정 이유:** 커뮤니티 크기 때문에 초기 단계에서 위험

### Consequences

**긍정적:**
- ✅ 풍부한 라이브러리 생태계 → 개발 속도 빠름
- ✅ 대량의 튜토리얼/Q&A → 문제 해결 빠름
- ✅ TypeScript 네이티브 지원 → 타입 안전성 높음
- ✅ Vite로 초고속 개발 환경 (HMR < 100ms)

**부작용/추후 과제:**
- ⚠️ 번들 사이즈가 Vue/Svelte보다 큼 → 번들 최적화 필요
  - **대응**: Code splitting (React.lazy), 라이브러리 검토
- ⚠️ 상태관리 선택 필요 (내장 안 됨)
  - **대응**: Zustand 사용 (간단하고 가벼움)
- ⚠️ 보일러플레이트 코드 많을 수 있음
  - **대응**: 컴포넌트 라이브러리(shadcn/ui) 활용

**검토 주기:** 번들 사이즈 모니터링 (매 주간 빌드)

## ADR-002: 백엔드 프레임워크 선택 (Hono + TypeScript)

**Status**: Accepted  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

백엔드 프레임워크를 선택해야 한다. 조건:

- **개발 속도**: 빠른 API 구현
- **타입 안전성**: 프론트엔드와 타입 공유 가능
- **배포 유연성**: Vercel Edge Functions 또는 전통 서버 모두 지원
- **성능**: 감정 기록 조회/저장의 낮은 지연시간
- **러닝커브**: 팀원이 빠르게 학습 가능

### Decision

**Hono + TypeScript를 선택한다.**

Hono는 TypeScript-first 설계로 프론트엔드와 타입을 공유할 수 있고, Vercel Edge Functions/Cloudflare Workers/Node.js 모두 지원하므로 배포 선택지가 유연하다. 또한 극도로 가볍고 빠르다.

### Alternatives

#### 1. Express.js

**장점:**
- 가장 널리 사용되는 Node.js 프레임워크
- 무수한 미들웨어와 플러그인
- 튜토리얼 많음

**단점:**
- 2010년대 설계로 구식 (callback-heavy)
- TypeScript 지원이 뒤처짐 (서드파티 타입 정의 필요)
- 번들 사이즈 큼
- Edge Functions 지원 약함

**결정 이유:** 구식 패러다임, TypeScript 지원 약함

#### 2. Python FastAPI

**장점:**
- 현대적이고 우아한 API 설계
- 자동 API 문서 생성 (OpenAPI)
- 높은 성능

**단점:**
- 프론트엔드와 타입 공유 어려움
- 팀원이 Python 숙련도 필요
- 배포 (Vercel, Cloudflare) 복잡
- Node.js 호스팅보다 설정 복잡

**결정 이유:** 프론트엔드와 타입 공유 불가, 배포 복잡

#### 3. Firebase (BaaS)

**장점:**
- 백엔드 개발 최소화
- 인증, 데이터베이스 통합
- 확장성 자동

**단점:**
- 벤더 락인 (Google에 종속)
- 복잡한 쿼리 구현 어려움
- 비용 증가 위험 (pay-per-call)
- 커스터마이징 제한

**결정 이유:** 벤더 락인 위험, 복잡한 데이터 쿼리 필요

### Consequences

**긍정적:**
- ✅ 프론트엔드와 동일 언어(TypeScript) → 코드 공유, 타입 안전성
- ✅ 매우 가벼운 번들 사이즈
- ✅ Edge Functions 지원 → 글로벌 배포 가능
- ✅ 현대적 async/await 문법

**부작용/추후 과제:**
- ⚠️ 커뮤니티 규모 Express보다 작음
  - **대응**: 기본 기능은 표준 HTTP/REST로 구현하고, 특수한 건 Hono 문서 참고
- ⚠️ 미들웨어 생태계 제한적
  - **대응**: 직접 구현하거나 npm 검색으로 대체
- ⚠️ 팀원이 Hono 학습 필요 (러닝커브 낮지만)
  - **대응**: 온보딩 문서 작성 예정

**검토 주기:** API 성능 모니터링 (매 배포 후)

## ADR-003: 데이터베이스 선택 (MySQL + Prisma ORM)

**Status**: Accepted (Updated 2026-07-08)  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

데이터베이스를 선택해야 한다. Self Mental의 요구사항:

- **보안**: 사용자의 개인 감정 데이터 → 애플리케이션 레벨 접근 제어
- **성능**: 기록 조회/저장의 낮은 지연시간
- **구조**: 감정 기록은 스키마가 구조화됨 (정규화된 데이터)
- **비용**: MVP 단계이므로 무료 또는 저비용
- **개발 속도**: 타입 안전성 + 자동 마이그레이션 관리

### Decision

**MySQL + Prisma ORM를 선택한다.**

로컬 개발과 초기 배포에 최적화된 구성으로, Prisma ORM은 TypeScript 타입 안전성을 제공하고 자동 마이그레이션 관리로 개발 속도를 높인다. 벤더 락인이 없어 향후 다른 데이터베이스로 전환이 자유롭다.

### Alternatives

#### 1. Supabase PostgreSQL (이전 선택)

**장점:**
- RLS 네이티브 지원
- PostgreSQL 강력한 쿼리
- 초기 무료 + 저비용

**단점:**
- 벤더 락인 (Supabase 종속)
- 초기 무료 이후 비용 증가
- 아직 배포 전이므로 불필요한 과기능

**변경 이유:** MVP 단계에서 Supabase 비용을 피하기 위해 로컬 MySQL로 간단하게 시작

#### 2. Firebase Firestore

**장점:**
- 백엔드 개발 최소화

**단점:**
- 벤더 락인 (Google)
- 복잡한 쿼리 제한
- 비용 예측 어려움

**결정 이유:** 복잡한 데이터 쿼리 필요, 벤더 락인

#### 3. PostgreSQL + ORM 없이

**장점:**
- 직접 제어 가능
- 원시 SQL 최적화

**단점:**
- 마이그레이션 관리 복잡
- SQL 문자열 작성 오류 가능
- 타입 안전성 낮음

**결정 이유:** Prisma ORM으로 타입 안전성 + 마이그레이션 관리 자동화

### Consequences

**긍정적:**
- ✅ 벤더 락인 없음 → 향후 다른 DB로 자유로운 전환
- ✅ Prisma 자동 마이그레이션 → 스키마 관리 편함
- ✅ TypeScript 타입 안전성 → 쿼리 컴파일 타임 검증
- ✅ 로컬 개발 비용 0원
- ✅ 초기 배포 단계에 최적화

**부작용/추후 과제:**
- ⚠️ 애플리케이션 레벨 접근 제어 필요
  - **대응**: 세션 토큰으로 user_id 검증 (현재 구현됨)
- ⚠️ 수평 확장 시 별도 전략 필요
  - **대응**: v1.1에서 필요 시 PostgreSQL 또는 Cloud SQL로 마이그레이션
- ⚠️ 백업 자동화 필요
  - **대응**: 로컬 개발 단계에서는 최소, 배포 후 자동 백업 구성

**검토 주기:** 첫 배포 후 데이터베이스 성능 모니터링 (월간)

## ADR-004: 9단계 구조 + 인지 왜곡 7가지 (vs 단순화)

**Status**: Accepted  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

감정 정리 과정의 복잡도를 결정해야 한다. 선택지:

1. **단순**: 상황 + 감정 + 느낌 (3단계)
2. **중간**: 우리가 선택한 9단계 + 인지 왜곡 7가지
3. **복잡**: CBT 전체 (10단계+)

요구사항:
- **사용자 이탈 최소화**: 입력이 너무 길면 포기
- **심리적 가치**: 단순 감정 기록을 넘어 객관적 사고로 전환
- **대학생 타겟**: 자책 심한 사용자에게 인지 왜곡 인식이 중요

### Decision

**9단계 구조 + 인지 왜곡 7가지를 선택한다.**

이는 CBT 이론과 실제 대학생 심리를 균형 맞춘 선택이다. 너무 단순하면 심리적 효과 떨어지고, 너무 복잡하면 사용자가 포기한다.

### Alternatives

#### 1. 3단계 (상황 → 감정 → 느낌)

**장점:**
- 진입 장벽 낮음
- 1-2분 안에 완료 가능
- 초기 사용자 유입 많을 것

**단점:**
- 심리적 개입 미흡 (단순 감정 일기)
- 사용자가 객관적 사고로 전환 안 됨
- "왜 쓰지?" → 이탈 확률 높음
- 경쟁 앱(Day One, 일기 앱)과 차별성 없음

**결정 이유:** 우리의 핵심 가치(객관적 사고 전환)를 실현 불가

#### 2. 12단계 (CBT 완전판)

**장점:**
- 더 정교한 심리 개입
- 심리 상담과 더 유사
- 장기 효과 더 클 수 있음

**단점:**
- 입력 시간 10분 이상
- 사용자 피로도 높음
- MVP 단계에서 개발 기간 2배
- 대학생 타겟에는 과부하

**결정 이유:** MVP 빠른 출시 우선, 사용자 이탈 위험

#### 3. 인지 왜곡 10가지 (vs 우리의 7가지)

**장점:**
- 더 많은 패턴 포괄
- 심리학적 완성도 높음

**단점:**
- 대학생 사용자에게 정보 과부하
- 선택 시간 증가
- 자주 안 쓰는 왜곡도 포함 (예: "읽은 마음")

**결정 이유:** 7가지는 대학생이 실제 경험하는 주요 패턴 (과잉일반화, 결론도약, 개인화, 확대/축소, 흑백사고, 라벨붙이기, 감정적추론)

### Consequences

**긍정적:**
- ✅ 단순한 감정 기록을 넘어 인지 변화 유도
- ✅ CBT 기반 검증된 방법 사용
- ✅ 대학생 심리에 맞춘 패턴 선별
- ✅ 적절한 복잡도 → 완성률 70% 이상 기대

**부작용/추후 과제:**
- ⚠️ 7가지 인지 왜곡이 실제 사용자와 맞지 않을 수 있음
  - **대응**: v1.1에서 사용자 피드백 기반 추가/삭제
- ⚠️ 일부 사용자는 여전히 9단계를 길다고 느낄 것
  - **대응**: 단계별 자동 저장, 임시 저장으로 중단 가능하게 설계
- ⚠️ 교육 필요 (인지 왜곡 개념 설명)
  - **대응**: 각 단계마다 가이드 텍스트, 예시 제공

**검토 주기:** 월간 완성률/포기율 추적, v1.1 인지 왜곡 확대 검토

## ADR-005: MVP 범위 (10단계, 검색, 통계 제외)

**Status**: Accepted  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

출시 범위를 결정해야 한다. 할 수 있는 모든 걸 하려면 8주 이상 필요하지만, MVP는 4주 목표다.

질문: "꼭 필요한 건 무엇인가?"

### Decision

**MVP에서 10단계(감정 변화 추적), 검색/필터링, 통계는 제외한다.**

4주 안에 1-9단계 핵심 기능을 안정적으로 출시하는 것이 우선이며, 사용자 피드백 수집 후 v1.1에서 추가하는 것이 효율적이다.

### Alternatives

#### 1. 10단계 포함 (모든 기능 한번에)

**장점:**
- 처음부터 완성도 높음
- "감정 변화 추적"은 매력적인 기능

**단점:**
- 개발 기간 8주 이상
- 출시 지연 → 사용자 피드백 늦음
- 버그 많을 가능성
- MVP 원칙 어김

**결정 이유:** 시간 제약, 사용자 피드백 우선

#### 2. 검색/필터링 포함

**장점:**
- 기록이 많아지면 필요한 기능
- 경쟁 앱과의 기능 대비

**단점:**
- 초기 사용자는 기록 5-10개만 있음 (검색 불필요)
- 검색 인덱싱 구현 복잡
- MVP 일정 연장
- "나중에 필요" vs "지금 필요"의 구분

**결정 이유:** 초기 사용자는 검색 필요 없음, 사용자 피드백으로 우선순위 정할 수 있음

#### 3. 통계 대시보드 포함

**장점:**
- 데이터 시각화로 사용자 재방문 유도
- 심리 앱으로서의 차별성

**단점:**
- 초기 데이터 부족 (의미 없음)
- 차트 라이브러리 추가
- 백엔드 집계 쿼리 복잡
- 개발 기간 2주 이상 추가

**결정 이유:** 초기 데이터 부족, v1.1 후 가치 있음

### Consequences

**긍정적:**
- ✅ 4주 MVP 출시 가능
- ✅ 빠른 사용자 피드백 수집
- ✅ 개발팀 번아웃 방지
- ✅ 가장 핵심 기능에 집중
- ✅ 명확한 v1.1 로드맵 가능

**부작용/추후 과제:**
- ⚠️ 초기 사용자가 "검색이 없네?"라고 느낄 것
  - **대응**: 기록이 적을 때는 필요 없다는 설명, 피드백 폼에서 요청 수집
- ⚠️ 감정 변화 추적을 기대했던 사용자의 실망
  - **대응**: 로드맵 공개, v1.1 출시 일정 안내
- ⚠️ 경쟁 앱(Moodpath, 우울증 앱)과의 기능 격차 발생 가능
  - **대응**: 우리는 "진정한 CBT 구조"에 집중, 기능 경쟁이 아닌 효과 입증

**검토 주기:** v1.1 계획 시 사용자 피드백 분석

## ADR-006: 배포 플랫폼 선택 (Vercel)

**Status**: Accepted  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

프론트엔드 + 백엔드를 배포할 플랫폼을 선택해야 한다.

조건:
- **자동 배포**: Git push 시 자동 배포
- **비용**: 초기 저비용
- **성능**: CDN, 지연시간 최소화
- **개발자 경험**: 설정 최소화 (zero-config)

### Decision

**Vercel을 선택한다.**

Hono + React + TypeScript 스택에서 Vercel이 최적이다. Git 통합이 자동이고, 초기 무료, 성능 우수하며, 특히 Next.js 같은 풀스택을 지원한다.

### Alternatives

#### 1. Railway

**장점:**
- 프론트엔드 + 백엔드 + DB 한 곳에서 관리
- 비용이 합리적
- PostgreSQL 통합

**단점:**
- Vercel만큼 자동화 안 됨
- CDN이 약함 (성능)
- 초기 설정 복잡
- 배포 속도 느림

**결정 이유:** 성능, 초기 설정 복잡도

#### 2. Cloudflare Workers

**장점:**
- 엣지 기반 초저지연
- 비용 매우 저렴
- Hono 최적화

**단점:**
- 개발자 경험 복잡 (wrangler CLI 필수)
- 데이터베이스 통합 약함
- 초기 러닝커브 높음
- 콜드스타트 고려 필요

**결정 이유:** 초기 개발 경험 우선, 성능은 Vercel도 충분

#### 3. AWS (EC2 + RDS)

**장점:**
- 완전한 제어
- 장기적으로 비용 효율적
- 엔터프라이즈 기능 풍부

**단점:**
- 초기 설정 매우 복잡
- DevOps 역량 필수
- 비용 관리 어려움
- MVP 단계에서 과잉

**결정 이유:** MVP 단계에서 운영 부담, 설정 복잡

### Consequences

**긍정적:**
- ✅ Git push → 자동 배포 (CI/CD 설정 불필요)
- ✅ 무료 SSL/TLS
- ✅ 글로벌 CDN (한국, 미국 모두 빠름)
- ✅ 초기 6개월 무료/저비용
- ✅ Vercel Analytics로 성능 모니터링

**부작용/추후 과제:**
- ⚠️ Vercel 벤더 락인 (마이그레이션 시 비용)
  - **대응**: 컨테이너 기반 구조로 설계 (Docker)
- ⚠️ 콜드스타트 (가끔 초기 요청 1-2초 지연)
  - **대응**: 정상 범위, 사용량 증가 시 자동 해결
- ⚠️ 환경변수 관리 학습 필요
  - **대응**: 문서 작성, 팀원 온보딩

**검토 주기:** 월간 배포 로그 확인, 분기별 성능 검토

## ADR-007: 상태 관리 라이브러리 선택 (Zustand)

**Status**: Accepted  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

React에서 복잡한 상태 관리가 필요하다:

- 사용자 인증 상태 (로그인, 로그아웃)
- 현재 작성 중인 기록 (1-9단계 폼 상태)
- 기록 목록 캐시
- 알림/토스트 상태

선택지: Context API, Redux, Zustand, Recoil, Jotai

### Decision

**Zustand를 선택한다.**

간단한 문법, 보일러플레이트 최소, 작은 번들 사이즈, 그리고 나중에 Redux로 마이그레이션 가능하기 때문이다.

### Alternatives

#### 1. React Context API (내장)

**장점:**
- 외부 라이브러리 불필요
- 제로 번들 사이즈 증가

**단점:**
- 깊은 nesting (props drilling)
- 리렌더링 성능 이슈
- 복잡한 로직 관리 어려움

**결정 이유:** 9단계 폼 상태 관리에는 Context API 부족

#### 2. Redux

**장점:**
- 산업 표준, 커뮤니티 큼
- DevTools로 디버깅 우수
- 강력한 미들웨어

**단점:**
- 보일러플레이트 매우 많음
- 초보자 학습곡선 높음
- 번들 사이즈 큼 (초기 앱에는 과잉)
- 초기 개발 속도 저하

**결정 이유:** MVP 단계에서 복잡도 높음, 추후 필요시 마이그레이션 가능

#### 3. Recoil / Jotai

**장점:**
- 원자적(atomic) 상태 관리
- 세밀한 재렌더링 제어

**단점:**
- 아직 실험적 (Facebook 지원이지만 명확한 로드맵 없음)
- 커뮤니티 규모 작음
- 초기 학습곡선 가파름

**결정 이유:** 안정성, 커뮤니티 규모

### Consequences

**긍정적:**
- ✅ 극도로 간단한 문법 (3줄의 store 정의)
- ✅ DevTools 지원 (Redux DevTools 호환)
- ✅ 번들 사이즈 작음 (~2KB)
- ✅ 팀원 학습 시간 최소

**부작용/추후 과제:**
- ⚠️ 상태가 복잡해지면 Redux로 마이그레이션 고려
  - **대응**: 모듈식 구조로 설계하면 마이그레이션 용이
- ⚠️ 미들웨어 기능 제한적
  - **대응**: 복잡한 로직은 custom hook이나 React Query로 분리

**검토 주기:** v1.1에서 상태 복잡도 평가 후 Redux 마이그레이션 검토

## ADR-008: 스타일링 솔루션 (Tailwind CSS)

**Status**: Accepted  
**Date**: 2026-07-02  
**Author**: 정우영

### Context

React UI를 스타일링하는 방식을 결정해야 한다.

요구사항:
- **일관성**: 디자인 시스템(design.md)과의 토큰 일치
- **개발 속도**: 빠른 스타일 작성
- **유지보수**: 변수/색상 중앙화 관리
- **성능**: 불필요한 CSS 제거 (tree-shaking)

### Decision

**Tailwind CSS를 선택한다.**

design.md와 DESIGN_TOKENS.md의 토큰을 Tailwind 클래스로 직접 매핑할 수 있으므로, 설계와 구현이 일치한다.

### Alternatives

#### 1. CSS Modules

**장점:**
- 스코프된 CSS (이름 충돌 없음)
- CSS 학습만으로 충분

**단점:**
- 클래스명 작성이 번거로움
- 디자인 토큰 일관성 유지 어려움
- 개발 속도 느림
- 반응형(media query) 작성 번거로움

**결정 이유:** 개발 속도, 디자인 일관성

#### 2. Styled-Components / Emotion (CSS-in-JS)

**장점:**
- 동적 스타일링 가능
- 자동 vendor prefix
- 컴포넌트 기반

**단점:**
- 런타임 오버헤드
- 번들 사이즈 증가
- SSR 복잡도 (필요 없음)
- Tailwind보다 개발 속도 느림

**결정 이유:** 동적 스타일 많지 않음, 성능

#### 3. Sass/Less (전처리기)

**장점:**
- 변수, 믹스인 등 기능 풍부
- 기존 CSS 지식으로 확장

**단점:**
- 빌드 단계 추가
- 사용하지 않는 스타일 최적화 어려움
- Tailwind보다 개발 속도 느림

**결정 이유:** 불필요한 CSS 제거, 개발 속도

### Consequences

**긍정적:**
- ✅ design.md 토큰과 1:1 대응 (일관성 보장)
- ✅ 반응형 빌트인 (sm:, md:, lg: 프리픽스)
- ✅ 극도로 빠른 개발 (클래스만 추가)
- ✅ 최종 번들에 사용하지 않는 CSS 제거
- ✅ 팀원 온보딩 빠름 (클래스명 암기하면 됨)

**부작용/추후 과제:**
- ⚠️ 동적 클래스 생성 주의
  - **대응**: `className={...}` 동적 생성 피하고, 미리 정의된 클래스만 사용
- ⚠️ HTML이 verbose해질 수 있음 (클래스 많음)
  - **대응**: 컴포넌트 분리로 관리, `@apply` 지시어 사용
- ⚠️ 복잡한 상황별 스타일 (조건부 많음)
  - **대응**: clsx 라이브러리 사용

**검토 주기:** 컴포넌트 개발 시 스타일 코드 리뷰
