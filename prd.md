## 1. 배경 및 문제 정의

### 1.1 배경

감정이 힘든 순간 사람들은 다양한 방식으로 대처한다.

- 친구에게 말하기 → 위로는 받지만, 내 생각이 객관적인지 점검 안 됨
- 명상/심리 영상 → 도움되지만, 감정이 올라오는 순간 적용 어려움
- 일기장 → 구조 없이 자책 기록으로 끝남

### 1.2 핵심 문제

**대학생/취업준비생이 경험하는 문제:**

- 팀플, 발표, 피드백에서 쉽게 불안/자책에 빠짐
- 상대의 말투를 부정적으로 해석하는 경향 → 오래 곱씹음
- "내 생각이 사실인가, 추측인가?" 정리해줄 누군가 필요함
- 상담은 부담스럽지만, 혼자 마음을 객관적으로 정리하고 싶음

### 1.3 Self Mental이 해결하는 것

사용자가 **9단계 구조 (상황→감정→생각→사실/해석→인지 왜곡→근거→대안→행동)**를 따라가며:

- ✓ **사실과 해석을 분리** → 객관적 사고로 전환
- ✓ **인지 왜곡 구체적 인식** → "내가 왜곡하고 있었구나" 깨달음
- ✓ **현실적인 대안과 행동** → 단순 감정 정리를 넘어 행동으로 연결

## 3. 대상 사용자

### 3.1 Primary User: 대학생/취업준비생

**인구통계:**
- 나이: 19-28세
- 교육: 대학 재학/졸업
- 기술 리터러시: 높음 (웹앱 사용 경험 충분)

**행동 특성:**
- 팀플, 발표, 피드백에서 자주 불안/자책 느낌
- 피드백받으면 오래 곱씹으며 반추 (rumination)
- 메모나 명상만으로는 생각 정리가 안 됨
- "이게 사실일까?" 누군가 객관적으로 정리해줬으면 함

**심리 특성:**
- 완벽주의 경향 (실수 → 자책)
- 인관관계 민감성 (상대의 반응 → 부정적 해석)
- 행동 변화 의욕 있음 (해결책을 찾고 싶음)

### 3.2 Secondary User: 일반 성인 (향후 확대)

- 직장 스트레스 관리 필요
- 자기 성찰 관심 있음

## 4. 핵심 기능

### 4.1 기능 목록

#### A. 사용자 인증 (필수)

- ✓ 이메일 + 비밀번호 회원가입/로그인
- ✓ 비밀번호 재설정 (이메일 기반)
- ✓ 로그아웃
- ○ 소셜 로그인 (Google, GitHub) - v1.1 고려

#### B. 감정 기록 작성 (1-9단계)

사용자가 다음 순서로 감정을 정리하고 저장할 수 있습니다.

| 단계 | 입력 형태 | 예시 |
|------|----------|------|
| **1단계: 상황** | 텍스트 (최대 1000자) | "팀플 회의에서 내 의견을 팀장이 무시한 것 같았다" |
| **2단계: 감정** | 선택 (1개) | 불안, 자책, 수치심, 후회, 분노, 좌절, 외로움 중 1개 |
| **3단계: 감정 강도** | 슬라이더 (0~100) | 75 (높음) |
| **4단계: 자동 생각** | 텍스트 (최대 500자) | "내 능력이 부족해서 무시당하는 거구나" |
| **5단계: 사실 vs 해석** | 텍스트 분리 | **사실**: 팀장이 내 의견에 응답 안 함 / **해석**: 내 능력 부족 때문 |
| **6단계: 인지 왜곡** | 다중선택 (최대 7개) | 과잉일반화, 결론도약, 개인화 |
| **7단계: 근거** | 텍스트 분리 | **찬성**: 지난 프로젝트에서도 피드백 못 받음 / **반대**: 다른 팀원도 같은 경험 |
| **8단계: 대안 생각** | 텍스트 (최대 500자) | "팀장이 바빴을 수도 있고, 내 의견 품질이 부족했을 가능성도 있다" |
| **9단계: 작은 행동** | 텍스트 (최대 200자) | "다음 회의 전에 의견을 정리해서 발표하기" |

#### C. 기록 관리

- ✓ **임시 저장**: 각 단계 완료 시 자동 저장 (초안 형태)
- ✓ **최종 저장**: 9단계 완료 후 기록 고정
- ✓ **수정**: 저장된 기록 다시 열어 수정 가능
- ✓ **삭제**: 기록 삭제 가능 (재확인 팝업 후)
- ✓ **조회**: 본인의 모든 기록 목록 조회 (최신순, 페이징)

#### D. 기록 목록 페이지

- 감정 배지 (색상+아이콘) + 감정명 + 날짜
- 기록 상태 배지 (작성 완료 / 임시 저장)
- 한 줄 요약 (상황의 첫 문장 또는 감정 강도)
- 클릭 시 기록 상세 보기

#### E. 마이페이지 (기본)

- 계정 정보 표시
- 로그아웃 버튼
- v1.1: 비밀번호 변경, 데이터 다운로드, 계정 삭제

### 4.2 사용자 여정

```
1. 로그인/회원가입
   ↓
2. 기록 목록 페이지 진입
   ↓
3. "새 기록 작성" 클릭
   ↓
4. 1단계 상황 입력 → "다음"
   ↓
5. 2단계 감정 선택 → "다음"
   ↓
6. ... (3-8단계 반복)
   ↓
9. 9단계 행동 입력 → "저장"
   ↓
10. 기록 목록으로 돌아가기 (완성된 기록 표시)
```

## 5. 성공 기준

### 5.1 기능 완성도

| 기능 | 완성 기준 | 검증 방법 |
|------|----------|----------|
| **회원가입/로그인** | 이메일 기반 인증 동작 | 수동 테스트 |
| **1-9단계 작성** | 모든 입력 필드 저장 가능 | 자동 테스트 + 사용자 테스트 |
| **임시 저장** | 작성 중 새로고침 후 내용 복구 | 사용자 테스트 |
| **기록 목록** | 사용자의 모든 기록 표시, 최신순 정렬 | 자동 테스트 |
| **기록 수정** | 저장된 기록 수정 가능 | 사용자 테스트 |
| **기록 삭제** | 재확인 후 삭제, 복구 불가 | 사용자 테스트 |

### 5.2 비기능 기준

| 요구사항 | 기준 | 검증 |
|----------|------|------|
| **페이지 로드** | 2초 이내 | Lighthouse, PageSpeed Insights |
| **API 응답** | 500ms 이내 | 개발 환경 테스트 |
| **모바일 반응형** | 375px~2560px 지원 | 실제 디바이스 테스트 |
| **보안** | HTTPS + RLS 정책 | 보안 체크리스트 |
| **가용성** | 99% (배포 서비스 보장) | 배포 후 모니터링 |

### 5.3 사용자 경험 기준

| 기준 | 목표 | 측정 |
|------|------|------|
| **기록 완성률** | 9단계 모두 작성하는 기록 > 70% | 데이터 분석 |
| **활성 사용자** | 주 3회 이상 기록하는 사용자 > 50% | DAU/WAU 추적 |
| **사용자 만족도** | NPS > 30 | 사용자 설문 |
| **인지 왜곡 활용** | 평균 2개 이상 선택 | 데이터 분석 |
| **앱 재방문율** | 7일 내 재방문 > 60% | 웹 분석 |

### 5.4 출시 체크리스트

배포 전 반드시 확인해야 할 항목들:

- [ ] 모든 페이지 모바일 테스트 완료
- [ ] 로그인/로그아웃 흐름 동작 확인
- [ ] 9단계 전체 작성 → 저장 → 조회 → 수정 → 삭제 각각 테스트
- [ ] 에러 상황 처리 (네트워크 끊김, API 실패)
- [ ] 성능 테스트 (Lighthouse > 80)
- [ ] 보안 테스트 (SQL injection, XSS, CSRF)
- [ ] RLS 정책 동작 확인 (다른 사용자의 기록 접근 불가)
- [ ] 배포 및 HTTPS 확인
- [ ] 모니터링 설정 (에러 로깅, 성능)

## 6. 제외 범위

### 6.1 MVP에서 제외하는 기능

| 기능 | 이유 | 예정 버전 |
|------|------|----------|
| **10단계: 감정 변화 추적** | 기록이 충분히 쌓여야 의미 있음 | v1.1 |
| **감정별 필터/검색** | 초기 사용자는 기록 수 적음, 구현 복잡 | v1.1 |

| **AI 추천 기능** | 모델 학습 데이터 부족 | v2.0+ |
| **소셜 로그인** | 이메일 인증으로 충분 | v1.1 |
| **End-to-End 암호화** | 복잡도 vs 가치, 표준 암호화로 충분 | v2.0+ |
| **오프라인 모드** | 인터넷 환경 가정 | 향후 |
| **다국어 지원** | 한국어만 | 향후 |

### 6.2 기술적 제외

- ❌ **실시간 멀티플레이어 편집**: 개인용 앱이므로 불필요
- ❌ **모바일 네이티브 앱**: 웹앱으로 충분 (향후 PWA 고려)
- ❌ **음성 입력**: 텍스트 입력이 정보량 많음
- ❌ **동기화 (여러 기기 간)**: MVP 단계에서는 단일 기기 가정

### 6.3 법률/정책 제외

- ❌ **의료/상담 서비스**: 교육/성찰 도구일 뿐, 전문가 상담 대체 불가
- ❌ **고급 개인정보보호**: GDPR 완전 준수 (v1.1), MVP는 기본 준수
- ❌ **광고**: 사용자 경험 방해 금지

## 7. 성공 사례 및 실패 사례

### 7.1 성공 시나리오

```
사용자 A (대학생)
1. 발표 피드백 후 불안감으로 가입
2. "새 기록" → 상황 입력 → ... → 9단계 완성
3. 자동 생각: "내 능력이 부족하다"
   → 사실/해석 분리: 사실(피드백 받음) vs 해석(능력 부족)
   → 인지 왜곡 선택: "과잉일반화" 깨달음
4. 대안 생각: "한 번의 실패가 전부는 아니다"
5. 행동: "다음 발표 전에 준비 시간 충분히 잡기"
6. 기록 저장 후 마음이 조금 가벼워짐
7. 다음 주 같은 상황에서 다시 기록
```

**결과**: 정기적 사용 → 자기 성찰 습관 → NPS 긍정적

### 7.2 실패 시나리오

```
사용자 B
1. 가입 후 첫 기록 시작
2. 3단계에서 포기 (입력 필드가 많음)
3. 기록 미완료
4. 다시 들어오지 않음
```

**원인**: 초기 진입장벽 (정보 과부하)  
**대응**: v1.1에서 단계 축약 또는 점진적 형식화 고려

## 11. 성공 정의

### MVP가 성공한다는 것

1. **기술적 성공**
   - 모든 핵심 기능이 안정적으로 동작
   - 성능 기준(2초 로드, 500ms API) 충족
   - 0일의 장애 시간

2. **사용자 성공**
   - 초기 10명 이상 가입
   - 최소 5명이 1-9단계 전체 완성
   - 3명 이상이 1주일 내 재방문

3. **비즈니스 성공**
   - 초기 피드백 수집 가능 (v1.1 방향 결정)
   - 사용자 만족도 NPS > 30
   - 개발팀이 확신 있게 v1.1 계획 가능

### 이것은 성공이 아님

- ❌ "출시했다" (기술 완료만으로는 불충분)
- ❌ "많은 사람이 가입했다" (실제 사용까지 봐야 함)
- ❌ "버그가 없다" (사용자 피드백이 더 중요)

## 12. 성공 측정 지표

### 핵심 지표 (OKR)

| O (목표) | KR (키 결과) | 담당 | 측정 주기 |
|----------|-------------|------|----------|
| 활성 사용자 확보 | MAU 10명 이상 | PM | 주간 |
| 기능 채택률 | 9단계 완성률 > 70% | 개발팀 | 주간 |
| 만족도 | NPS > 30 | PM | 2주간 |
| 재방문율 | WAU/MAU > 60% | 데이터 | 주간 |

### 추적 방법

- **Google Analytics 4**: 페이지뷰, 사용자 행동
- **Supabase**: 기록 생성/수정/삭제 로그
- **수동 설문**: 사용자 만족도, NPS
- **모니터링**: Vercel Analytics (성능), Sentry (에러)

## 13. 부록: 용어 정의

| 용어 | 정의 |
|------|------|
| **인지 왜곡** | 자동으로 떠오르는 부정적 생각 패턴 (CBT 이론) |
| **임시 저장** | 작성 중인 기록을 자동으로 저장 (草稿 상태) |
| **완성 저장** | 9단계 모두 작성 후 최종 저장 |
| **RLS** | Row Level Security - 데이터베이스에서 사용자별 접근 제어 |
| **NPS** | Net Promoter Score - 사용자 만족도 지표 (-100~100) |
| **DAU/MAU** | Daily/Monthly Active Users |
| **WAU** | Weekly Active Users |
